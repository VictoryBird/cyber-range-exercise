## 1. 개요

### 1.1 자산 목적

메일 포털 게이트는 발도리아(Valdoria) 정부 기관의 공개 DMZ 구간에 배치되는 **Nginx 리버스 프록시 서버**이다. 외부 인터넷에서 `webmail.mois.valdoria.gov` 도메인으로 접근하는 웹메일 트래픽을 수신하여, 내부망(INT) 192.168.92.205에서 운영 중인 Roundcube 웹메일 서버로 전달하는 단일 목적의 게이트웨이 역할을 수행한다.

### 1.2 훈련 시나리오 내 역할

- **레드팀 관점**: 공격 체인 STEP 1-2에 해당하며, 외부에서 내부망 진입을 위한 **첫 번째 피벗 포인트**로 활용된다. 프록시 헤더 조작, 경로 우회, Host 헤더 변조를 통해 내부 웹메일 서버에 직접 접근하거나 인증을 우회하는 공격을 수행한다.
- **블루팀 관점**: DMZ 구간에서 비정상적인 프록시 요청 패턴을 탐지하는 **초기 방어 거점**이다. Nginx 접근/오류 로그 분석을 통해 공격 징후를 식별해야 한다.

### 1.3 공격 체인 내 위치

```
[외부 공격자]
     │
     ▼
 ┌─────────────────────────────────┐
 │  OPNSense-1 (외부 방화벽)        │
 │  공인 IP → DMZ 포워딩            │
 └────────────┬────────────────────┘
              │ :80, :443
              ▼
 ┌─────────────────────────────────┐
 │  ★ 메일 포털 게이트 (본 자산)    │  ◀── STEP 1-2: 프록시 헤더/경로 조작
 │  192.168.92.202                  │
 │  Nginx Reverse Proxy             │
 └────────────┬────────────────────┘
              │ proxy_pass :8080
              ▼
 ┌─────────────────────────────────┐
 │  OPNSense-2 (내부 방화벽)        │
 └────────────┬────────────────────┘
              │
              ▼
 ┌─────────────────────────────────┐
 │  INT 웹메일 (Roundcube)          │  ◀── STEP 3 이후: 내부 공격 확장
 │  192.168.92.205                  │
 └─────────────────────────────────┘
```

### 3.2 요청 처리 흐름

```
1. 클라이언트 → webmail.mois.valdoria.gov:443 (HTTPS 요청)
2. Nginx SSL Termination → 평문 HTTP로 변환
3. location / → proxy_pass http://192.168.92.205:8080
4. Roundcube 응답 → Nginx → 클라이언트로 SSL 암호화 전달

* HTTP(:80) 요청은 HTTPS(:443)로 301 리다이렉트
```

## 6. Nginx 설정 명세

### 6.1 메인 설정 (nginx.conf)

```nginx
# /etc/nginx/nginx.conf

user www-data;
worker_processes 2;
pid /run/nginx.pid;
include /etc/nginx/modules-enabled/*.conf;

events {
    worker_connections 512;
}

http {
    # 기본 설정
    sendfile on;
    tcp_nopush on;
    types_hash_max_size 2048;

    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # 로깅
    log_format detailed '$remote_addr - $remote_user [$time_local] '
                        '"$request" $status $body_bytes_sent '
                        '"$http_referer" "$http_user_agent" '
                        '"$http_x_forwarded_for" "$http_x_real_ip" '
                        '"$host"';

    access_log /var/log/nginx/access.log detailed;
    error_log /var/log/nginx/error.log warn;

    # 타임아웃
    keepalive_timeout 65;
    proxy_connect_timeout 10;
    proxy_read_timeout 60;

    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}
```

### 6.2 사이트 설정 — 의도적 취약 버전 (webmail.conf)

> **[경고] 이 설정은 훈련용 의도적 취약점이 포함되어 있음**

```nginx
# /etc/nginx/sites-available/webmail.conf
# ★ 의도적 취약 설정 — 훈련 환경 전용

# ── HTTP → HTTPS 리다이렉트 ──
server {
    listen 80;
    server_name webmail.mois.valdoria.gov;
    return 301 https://$host$request_uri;
}

# ── HTTPS 메인 서버 ──
server {
    listen 443 ssl;

    # ★ 취약점 #1: server_name에 와일드카드 / 기본값 허용
    # 임의의 Host 헤더도 이 서버 블록에서 처리됨
    server_name webmail.mois.valdoria.gov _;

    # SSL 설정
    ssl_certificate /etc/nginx/ssl/webmail.crt;
    ssl_certificate_key /etc/nginx/ssl/webmail.key;
    include /etc/nginx/snippets/ssl-params.conf;

    # 접근 로그 (상세)
    access_log /var/log/nginx/webmail_access.log detailed;

    # ★ 취약점 #2: 클라이언트 제공 헤더를 검증 없이 백엔드로 전달
    # 공격자가 X-Forwarded-For, X-Real-IP를 조작하여 내부 IP로 위장 가능
    proxy_set_header X-Forwarded-For $http_x_forwarded_for;
    proxy_set_header X-Real-IP $http_x_real_ip;
    proxy_set_header Host $http_host;
    proxy_set_header X-Forwarded-Proto $scheme;

    # ── 메인 프록시 ──
    location / {
        proxy_pass http://192.168.92.205:8080;
    }

    # ★ 취약점 #3: 내부 관리 경로가 접근 제어 없이 프록시됨
    location /internal/ {
        proxy_pass http://192.168.92.205:8080/internal/;
    }

    # ★ 취약점 #4: 관리자 경로도 접근 제어 없이 노출
    location /admin/ {
        proxy_pass http://192.168.92.205:8080/admin/;
    }

    # ★ 취약점 #5: merge_slashes off로 경로 정규화 우회 허용
    merge_slashes off;

    # 에러 페이지
    error_page 502 503 504 /50x.html;
    location = /50x.html {
        root /usr/share/nginx/html;
    }
}
```

### 6.3 사이트 설정 — 보안 강화 버전 (참고용)

```nginx
# /etc/nginx/sites-available/webmail-secure.conf
# 보안 강화 참고 설정 — 실제 운영 환경 권장

server {
    listen 80;
    server_name webmail.mois.valdoria.gov;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;

    # [보안] 정확한 도메인만 허용, 기본 서버 블록과 분리
    server_name webmail.mois.valdoria.gov;

    ssl_certificate /etc/nginx/ssl/webmail.crt;
    ssl_certificate_key /etc/nginx/ssl/webmail.key;
    include /etc/nginx/snippets/ssl-params.conf;

    access_log /var/log/nginx/webmail_access.log detailed;

    # [보안] 프록시 헤더를 서버가 직접 설정 — 클라이언트 입력 무시
    proxy_set_header X-Forwarded-For $remote_addr;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header Host webmail.mois.valdoria.gov;
    proxy_set_header X-Forwarded-Proto $scheme;

    # [보안] merge_slashes 기본값(on) 유지로 경로 정규화
    # merge_slashes on;  (기본값이므로 명시 불필요)

    # [보안] 허용 경로만 명시적으로 프록시
    location / {
        # 내부 경로 패턴 차단
        if ($request_uri ~* "(\.\.|\%2e\%2e|/internal/|/admin/)") {
            return 403;
        }
        proxy_pass http://192.168.92.205:8080;
    }

    # [보안] 내부/관리 경로 명시적 차단
    location /internal/ {
        deny all;
        return 403;
    }

    location /admin/ {
        # 관리 대역에서만 접근 허용
        allow 192.168.92.0/24;
        deny all;
        proxy_pass http://192.168.92.205:8080/admin/;
    }

    # [보안] 알 수 없는 Host 헤더 차단을 위한 기본 서버 블록
    # (별도 default_server 블록에서 처리)

    error_page 502 503 504 /50x.html;
    location = /50x.html {
        root /usr/share/nginx/html;
    }
}

# [보안] 잘못된 Host 헤더 요청 차단용 기본 서버
server {
    listen 443 ssl default_server;
    server_name _;

    ssl_certificate /etc/nginx/ssl/webmail.crt;
    ssl_certificate_key /etc/nginx/ssl/webmail.key;

    return 444;  # 연결 즉시 종료
}
```

### 6.4 공통 스니펫

**ssl-params.conf:**

```nginx
# /etc/nginx/snippets/ssl-params.conf

ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers HIGH:!aNULL:!MD5:!RC4;
ssl_prefer_server_ciphers on;
ssl_session_timeout 1d;
ssl_session_cache shared:SSL:10m;
ssl_session_tickets off;

# HSTS (훈련 환경이므로 짧은 max-age)
add_header Strict-Transport-Security "max-age=3600" always;
```

**proxy-params.conf:**

```nginx
# /etc/nginx/snippets/proxy-params.conf

proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
proxy_buffering on;
proxy_buffer_size 4k;
proxy_buffers 8 4k;
```

## 7. 사용자/계정 설정

본 자산은 Nginx 리버스 프록시 전용이므로 애플리케이션 수준의 사용자 계정 관리가 없다. 사용자 인증은 백엔드 Roundcube(192.168.92.205)에서 처리된다.

### 7.1 시스템 계정

| 계정 | 용도 | 비고 |
|------|------|------|
| root | 시스템 관리 | 초기 설정 시에만 사용 |
| sysadmin | SSH 관리 접속 | sudo 권한, 키 기반 인증 권장 |
| www-data | Nginx 워커 프로세스 | 시스템 계정, 로그인 불가 |

### 7.2 참고사항

- 웹메일 로그인은 Roundcube 백엔드를 통해 처리되므로 본 프록시에서는 인증 로직이 없음
- 이 구조적 특성으로 인해 프록시 계층에서의 세션 관리가 부재하며, 이는 의도적 취약점의 일부임

### 8.2 VULN-02-01: 프록시 헤더 신뢰 취약점

**취약 설정 위치:** `/etc/nginx/sites-available/webmail.conf` 내 `proxy_set_header` 지시문

**취약 설정:**

```nginx
# 클라이언트가 보낸 헤더 값을 그대로 백엔드에 전달
proxy_set_header X-Forwarded-For $http_x_forwarded_for;
proxy_set_header X-Real-IP $http_x_real_ip;
proxy_set_header Host $http_host;
```

**문제점 상세:**

- `$http_x_forwarded_for`: 클라이언트의 HTTP 요청에 포함된 `X-Forwarded-For` 헤더 값을 검증 없이 백엔드로 전달한다. 공격자가 `X-Forwarded-For: 127.0.0.1` 또는 `X-Forwarded-For: 192.168.92.205`을 삽입하면 백엔드(Roundcube)는 요청이 내부에서 온 것으로 판단할 수 있다.
- `$http_x_real_ip`: 동일하게 클라이언트 조작 가능. 백엔드의 IP 기반 접근 제어를 우회하는 데 사용된다.
- `$http_host`: 클라이언트가 보낸 Host 헤더를 그대로 전달하므로, 백엔드가 가상 호스트 라우팅에 이 값을 사용할 경우 의도하지 않은 동작이 발생한다.

**공격 시나리오:**

```bash
# 공격 예시 1: 내부 IP 위장으로 Roundcube 관리 인터페이스 접근
curl -k https://webmail.mois.valdoria.gov/admin/ \
  -H "X-Forwarded-For: 127.0.0.1" \
  -H "X-Real-IP: 127.0.0.1"

# 공격 예시 2: 내부 네트워크 대역 위장
curl -k https://webmail.mois.valdoria.gov/ \
  -H "X-Forwarded-For: 192.168.92.205" \
  -H "X-Real-IP: 192.168.92.205"
```

**보안 대응 (올바른 설정):**

```nginx
# Nginx가 실제 클라이언트 IP를 직접 설정
proxy_set_header X-Forwarded-For $remote_addr;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header Host webmail.mois.valdoria.gov;  # 고정값
```

### 8.3 VULN-02-02: 경로 정규화 우회 (Path Traversal)

**취약 설정 위치:** `webmail.conf` 내 `merge_slashes off` 및 location 블록의 정규화 부재

**취약 설정:**

```nginx
# merge_slashes off로 이중 슬래시, 인코딩된 경로 구분자 허용
merge_slashes off;

# location 블록에 경로 검증 로직 없음
location / {
    proxy_pass http://192.168.92.205:8080;
}
```

**문제점 상세:**

- `merge_slashes off`가 설정되면 Nginx는 `//`, `///` 등의 다중 슬래시를 정규화하지 않는다.
- URL 인코딩된 경로 구분자(`%2F` = `/`, `%2e` = `.`)가 디코딩 전에 location 매칭을 수행하므로, `..%2Finternal/`과 같은 패턴이 `location /` 블록에 매칭된 후 백엔드에서 `../internal/`로 디코딩되어 경로 탈출이 가능하다.

**공격 시나리오:**

```bash
# 공격 예시 1: URL 인코딩을 이용한 내부 경로 접근
curl -k "https://webmail.mois.valdoria.gov/..%2Finternal/config"

# 공격 예시 2: 이중 인코딩을 이용한 관리 경로 접근
curl -k "https://webmail.mois.valdoria.gov/%2e%2e/admin/settings"

# 공격 예시 3: 다중 슬래시를 이용한 경로 혼동
curl -k "https://webmail.mois.valdoria.gov///internal/../admin/"

# 공격 예시 4: 혼합 인코딩
curl -k "https://webmail.mois.valdoria.gov/mail/..%2F..%2Finternal/"
```

**보안 대응:**

```nginx
# merge_slashes on (기본값 유지)
# location 블록에 경로 검증 추가
location / {
    if ($request_uri ~* "(\.\.|\%2e\%2e|\%2f|/internal/|/admin/)") {
        return 403;
    }
    proxy_pass http://192.168.92.205:8080;
}
```

### 8.4 VULN-02-03: Host 헤더 라우팅 조작

**취약 설정 위치:** `webmail.conf` 내 `server_name` 지시문

**취약 설정:**

```nginx
# 와일드카드(_)가 포함되어 모든 Host 헤더를 수용
server_name webmail.mois.valdoria.gov _;

# 클라이언트의 Host 헤더를 그대로 백엔드에 전달
proxy_set_header Host $http_host;
```

**문제점 상세:**

- `server_name`에 `_` (와일드카드)가 포함되어 있어 `Host: mail.mois.local`, `Host: 192.168.92.205` 등 임의의 Host 헤더 요청도 이 서버 블록에서 처리된다.
- `$http_host`를 통해 클라이언트가 지정한 Host 값이 백엔드 Roundcube로 그대로 전달되므로, 백엔드에서 Host 기반 가상 호스트 라우팅을 사용하는 경우 의도하지 않은 가상 호스트에 접근할 수 있다.
- 특히 `Host: mail.mois.local`과 같은 내부 도메인명을 사용하면 백엔드가 내부 관리 인터페이스를 응답할 가능성이 있다.

**공격 시나리오:**

```bash
# 공격 예시 1: 내부 도메인명으로 접근 시도
curl -k https://192.168.92.202/ \
  -H "Host: mail.mois.local"

# 공격 예시 2: 직접 내부 IP를 Host로 지정
curl -k https://192.168.92.202/ \
  -H "Host: 192.168.92.205"

# 공격 예시 3: localhost로 위장
curl -k https://webmail.mois.valdoria.gov/ \
  -H "Host: localhost"
```

**보안 대응:**

```nginx
# 정확한 도메인만 서버 블록에서 처리
server_name webmail.mois.valdoria.gov;

# Host 헤더를 고정값으로 설정
proxy_set_header Host webmail.mois.valdoria.gov;

# 잘못된 Host 요청은 별도 기본 서버 블록에서 차단
server {
    listen 443 ssl default_server;
    server_name _;
    return 444;
}
```

### 8.5 VULN-02-04: 내부 경로 무단 노출

**취약 설정 위치:** `webmail.conf` 내 `/internal/` 및 `/admin/` location 블록

**취약 설정:**

```nginx
# 접근 제어 없이 내부 관리 경로를 프록시
location /internal/ {
    proxy_pass http://192.168.92.205:8080/internal/;
}

location /admin/ {
    proxy_pass http://192.168.92.205:8080/admin/;
}
```

**문제점 상세:**

- Roundcube 백엔드의 `/internal/` 경로(내부 설정 인터페이스)와 `/admin/` 경로(관리 인터페이스)가 별도 location 블록으로 정의되어 있으나, `deny`, `allow`, `auth_basic` 등의 접근 제어가 전혀 없다.
- 외부 공격자가 `https://webmail.mois.valdoria.gov/internal/` 또는 `/admin/`에 직접 접근하면 백엔드의 관리 기능이 노출된다.
- 이 취약점은 VULN-02-01(헤더 조작)과 결합하면 관리 인터페이스의 IP 기반 접근 제어까지 우회할 수 있어 위험도가 증폭된다.

**공격 시나리오:**

```bash
# 공격 예시 1: 내부 설정 인터페이스 직접 접근
curl -k https://webmail.mois.valdoria.gov/internal/

# 공격 예시 2: 관리 인터페이스 접근 + 헤더 조작 결합
curl -k https://webmail.mois.valdoria.gov/admin/ \
  -H "X-Forwarded-For: 127.0.0.1" \
  -H "X-Real-IP: 127.0.0.1"

# 공격 예시 3: 경로 우회와 결합
curl -k "https://webmail.mois.valdoria.gov/..%2Finternal/config.php"
```

**보안 대응:**

```nginx
location /internal/ {
    deny all;
    return 403;
}

location /admin/ {
    allow 192.168.92.0/24;  # DMZ 관리 대역만 허용
    deny all;
    auth_basic "Admin Area";
    auth_basic_user_file /etc/nginx/.htpasswd;
    proxy_pass http://192.168.92.205:8080/admin/;
}
```

## 9. 블루팀 탐지 포인트

### 9.1 로그 소스

| 로그 파일 | 경로 | 내용 |
|----------|------|------|
| Nginx Access Log | `/var/log/nginx/access.log` | 기본 접근 로그 |
| Webmail Access Log | `/var/log/nginx/webmail_access.log` | 상세 포맷 (헤더 포함) |
| Nginx Error Log | `/var/log/nginx/error.log` | 오류 및 경고 |

### 9.2 로그 포맷 내 핵심 필드

상세 로그 포맷(`detailed`)에는 다음 필드가 포함된다:

```
$remote_addr          - 실제 클라이언트 IP
$http_x_forwarded_for - 클라이언트가 보낸 X-Forwarded-For (조작 가능)
$http_x_real_ip       - 클라이언트가 보낸 X-Real-IP (조작 가능)
$host                 - 요청의 Host 헤더 값
$request              - 전체 요청 라인 (메서드 + URI + 프로토콜)
```

### 9.3 탐지 지표 (IoC)

#### IoC-01: 내부 경로 직접 접근 시도

```
탐지 대상: /internal/, /admin/ 경로에 대한 외부 요청
로그 패턴: "GET /internal/" 또는 "GET /admin/"
심각도: HIGH
```

**grep 기반 탐지 쿼리:**

```bash
# 내부/관리 경로 직접 접근 탐지
grep -E '"(GET|POST|PUT|DELETE) /(internal|admin)/' /var/log/nginx/webmail_access.log

# URL 인코딩된 경로 탈출 시도 탐지
grep -iE '(%2e%2e|%2f|\.\./)' /var/log/nginx/webmail_access.log
```

#### IoC-02: Host 헤더 변조

```
탐지 대상: 정상 도메인이 아닌 Host 헤더 사용
정상 값: webmail.mois.valdoria.gov
비정상 값: mail.mois.local, 192.168.92.205, localhost, IP 주소 직접 사용
심각도: MEDIUM
```

**grep 기반 탐지 쿼리:**

```bash
# 비정상 Host 헤더 탐지 (정상 도메인이 아닌 모든 요청)
grep -v '"webmail.mois.valdoria.gov"$' /var/log/nginx/webmail_access.log | \
  grep -v '^#'

# 내부 도메인/IP를 Host로 사용하는 요청
grep -E '"(mail\.mois\.local|192\.168\.|127\.0\.0\.1|localhost)"' \
  /var/log/nginx/webmail_access.log
```

#### IoC-03: 비정상 프록시 헤더 삽입

```
탐지 대상: X-Forwarded-For 또는 X-Real-IP에 내부 IP 삽입
비정상 값: 127.0.0.1, 192.168.x.x, 10.x.x.x, 172.16-31.x.x
심각도: HIGH
```

**grep 기반 탐지 쿼리:**

```bash
# X-Forwarded-For에 내부 IP가 포함된 요청
grep -E '"(127\.0\.0\.1|192\.168\.[0-9]+\.[0-9]+|10\.[0-9]+\.[0-9]+\.[0-9]+)' \
  /var/log/nginx/webmail_access.log

# X-Forwarded-For 헤더가 존재하는 모든 요청 (외부에서 직접 접속 시 헤더가 없어야 정상)
grep -v '"- "' /var/log/nginx/webmail_access.log | grep 'X-Forwarded-For'
```

#### IoC-04: 경로 조작 시도

```
탐지 대상: URL 인코딩된 디렉토리 탈출 패턴
패턴: ..%2F, %2e%2e, %2e%2e%2f, ///, ../ 변형
심각도: HIGH
```

**grep 기반 탐지 쿼리:**

```bash
# 경로 탈출 패턴 탐지
grep -iE '(\.\./|\.\.%2[fF]|%2[eE]%2[eE]|%2[eE]\.|\.%2[eE]|/{3,})' \
  /var/log/nginx/webmail_access.log

# Nginx error.log에서 관련 오류
grep -E '(directory traversal|invalid|upstream)' /var/log/nginx/error.log
```

### 9.4 종합 모니터링 스크립트 (참고)

```bash
#!/bin/bash
# /opt/monitor/check_nginx_ioc.sh
# 블루팀 모니터링 참고 스크립트

LOG="/var/log/nginx/webmail_access.log"
ALERT_LOG="/var/log/nginx/security_alerts.log"

echo "=== [$(date)] Nginx IoC Check ===" >> "$ALERT_LOG"

# IoC-01: 내부 경로 접근
COUNT=$(grep -cE '"(GET|POST) /(internal|admin)/' "$LOG" 2>/dev/null)
[ "$COUNT" -gt 0 ] && echo "[HIGH] Internal path access: $COUNT hits" >> "$ALERT_LOG"

# IoC-02: Host 변조
COUNT=$(grep -cvE '"webmail\.mois\.valdoria\.gov"$' "$LOG" 2>/dev/null)
[ "$COUNT" -gt 5 ] && echo "[MEDIUM] Host header anomaly: $COUNT hits" >> "$ALERT_LOG"

# IoC-03: 헤더 조작
COUNT=$(grep -cE '"(127\.0\.0\.1|192\.168\.)' "$LOG" 2>/dev/null)
[ "$COUNT" -gt 0 ] && echo "[HIGH] Proxy header spoofing: $COUNT hits" >> "$ALERT_LOG"

# IoC-04: 경로 조작
COUNT=$(grep -ciE '(%2e|%2f|\.\.)' "$LOG" 2>/dev/null)
[ "$COUNT" -gt 0 ] && echo "[HIGH] Path traversal attempt: $COUNT hits" >> "$ALERT_LOG"
```

### 9.5 탐지 타임라인 (예상)

```
T+0min   공격자가 /admin/, /internal/ 경로 탐색 시작
         → IoC-01 트리거
T+5min   X-Forwarded-For: 127.0.0.1 헤더 삽입 시도
         → IoC-03 트리거
T+10min  Host: mail.mois.local 헤더로 변조된 요청
         → IoC-02 트리거
T+15min  ..%2F 패턴을 이용한 경로 우회 시도
         → IoC-04 트리거
T+20min  복합 공격: 헤더 조작 + 경로 우회 + 내부 경로 접근
         → IoC-01, 03, 04 동시 트리거
```

### 10.2 setup.sh 설계

```bash
#!/bin/bash
# assets/02_메일포털게이트/setup.sh
# 메일 포털 게이트 원클릭 배포 스크립트

set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

# .env 파일 로드
if [ -f "$SCRIPT_DIR/.env" ]; then
    source "$SCRIPT_DIR/.env"
else
    echo "[ERROR] .env 파일이 없습니다. .env.example을 복사하여 .env를 생성하세요."
    exit 1
fi

echo "=========================================="
echo " 메일 포털 게이트 배포 시작"
echo " IP: ${SERVER_IP}"
echo " Domain: ${DOMAIN}"
echo "=========================================="

# ── 1. 시스템 업데이트 ──
echo "[1/7] 시스템 패키지 업데이트..."
apt-get update -qq
apt-get upgrade -y -qq

# ── 2. Nginx 설치 ──
echo "[2/7] Nginx 설치..."
apt-get install -y -qq nginx openssl

# ── 3. 네트워크 설정 ──
echo "[3/7] 네트워크 인터페이스 설정..."
cat > /etc/netplan/01-netcfg.yaml <<NETPLAN
network:
  version: 2
  ethernets:
    eth0:
      addresses:
        - ${SERVER_IP}/24
      routes:
        - to: default
          via: ${GATEWAY_IP}
      nameservers:
        addresses: [8.8.8.8, 8.8.4.4]
NETPLAN
netplan apply || true

# ── 4. TLS 인증서 생성 ──
echo "[4/7] 자체 서명 TLS 인증서 생성..."
mkdir -p /etc/nginx/ssl
openssl req -x509 -nodes -days 365 \
    -newkey rsa:2048 \
    -keyout /etc/nginx/ssl/webmail.key \
    -out /etc/nginx/ssl/webmail.crt \
    -subj "/CN=${DOMAIN}/O=MOIS Valdoria/C=VD" \
    2>/dev/null
chmod 600 /etc/nginx/ssl/webmail.key

# ── 5. Nginx 설정 배포 ──
echo "[5/7] Nginx 설정 파일 배포..."
cp "$SCRIPT_DIR/conf/nginx/nginx.conf" /etc/nginx/nginx.conf
cp "$SCRIPT_DIR/conf/nginx/snippets/"*.conf /etc/nginx/snippets/
mkdir -p /etc/nginx/conf.d
cp "$SCRIPT_DIR/conf/nginx/conf.d/"*.conf /etc/nginx/conf.d/ 2>/dev/null || true

# 사이트 설정 배포
cp "$SCRIPT_DIR/conf/nginx/sites-available/webmail.conf" \
   /etc/nginx/sites-available/webmail.conf

# 기본 사이트 비활성화, 웹메일 사이트 활성화
rm -f /etc/nginx/sites-enabled/default
ln -sf /etc/nginx/sites-available/webmail.conf /etc/nginx/sites-enabled/webmail.conf

# 환경변수 치환 (백엔드 IP 등)
sed -i "s|__BACKEND_IP__|${BACKEND_IP}|g" /etc/nginx/sites-available/webmail.conf
sed -i "s|__BACKEND_PORT__|${BACKEND_PORT}|g" /etc/nginx/sites-available/webmail.conf
sed -i "s|__DOMAIN__|${DOMAIN}|g" /etc/nginx/sites-available/webmail.conf

# ── 6. 설정 검증 및 서비스 시작 ──
echo "[6/7] Nginx 설정 검증..."
nginx -t

echo "[6/7] Nginx 서비스 활성화 및 시작..."
systemctl enable nginx
systemctl restart nginx

# ── 7. 배포 확인 ──
echo "[7/7] 배포 확인..."
systemctl is-active nginx && echo "[OK] Nginx 실행 중" || echo "[FAIL] Nginx 시작 실패"
curl -sk https://localhost/ -o /dev/null -w "HTTPS 응답 코드: %{http_code}\n" || true

echo ""
echo "=========================================="
echo " 배포 완료"
echo " 접속: https://${DOMAIN}"
echo " 로그: /var/log/nginx/webmail_access.log"
echo "=========================================="
```

### 10.4 systemd 서비스 확인

Nginx는 Ubuntu 패키지 설치 시 systemd 유닛이 자동 생성된다. 필요시 override를 통해 커스터마이징한다:

```ini
# /etc/systemd/system/nginx.service.d/override.conf
[Service]
# 재시작 정책
Restart=always
RestartSec=5

# 로그 출력
StandardOutput=journal
StandardError=journal

# 리소스 제한 (훈련 환경)
LimitNOFILE=4096
```
